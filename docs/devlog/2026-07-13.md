### What Was Done
- **Root-cause exploration** (two parallel explore passes over the editor and check-mode/consistency surfaces). Five of Gaylon's seven reported macOS bugs reduce to two mechanisms:
  - The M4 `.onMove` drag recognizer is armed on every row at rest → ~0.5s click-to-highlight latency and flaky double-click rename. This is exactly the V1-4/D10 gate the M4 spec predicted; the pre-authored hover-gating fix was never applied.
  - Fragile `.onAppear { focused = true }` focus grabs → keystrokes fall through to the sidebar's `.searchable` (the "typing triggers search" mystery). Plus: `focusedCommandActions` closures capture `selectedID` by value at publish time → ⌘[/⌘] act on the last-added row.
  - Also confirmed: check mode has **zero** keyboard support (no selection model at all — the unfixed half of the 07-07 testing note), multi-select drag is a deliberate-but-silent no-op, the ellipsis menu hides shortcut hints because its target row may not equal the selection.
- **User decisions locked**: unify edit/check modes into ONE view (checkbox always visible, All/Remaining/Completed filter replaces the mode switch); Things 3 is the feel benchmark; **Return = rename** (Things-style). Consequence cascade: ⌘N = New Item (Things parity), New List → ⇧⌘N, ⌥⌘N = Add Above (bare ⇧Return dropped), ⌘E = menu-visible rename twin, ⌘K = Toggle Checked, Space = toggle check, Esc also clears selection, ⇧⌘E retired.
- **Spec written**: `docs/2026-07-13-m5-unified-editor-spec.md` (Rev 2) — complete interaction contract, all B1–B7 fixes, `CommandCatalog` registry-lite (menus + help doc render from one source; help is *generated*, so it cannot drift), deletion list (CheckModeView/CheckRowView/isCheckMode die), 4 phases each with a manual acceptance gate, file-by-file map, test migration map, ranked risks.
- **Interactive prototype built and published** (Claude artifact `e607fd31…`): a working simulation, not static mockups — macOS window (menu bar, toolbar, outline, inspector, context menus, full keyboard map, hover-armed drag, confirmation dialogs) and iOS iPhone frame (row-density check, tap/long-press/add-flow). Right rail: live keyboard legend + gate-matching "try this" script.

### Key Technical Insight
The M4 drag work shipped with its own predicted regression unverified: `.onMove` arms a drag recognizer on every row and delays click recognition on macOS — the spec's D10 hover-gating contingency existed but was never applied because the V1-4 manual gate was skipped. Lesson: a specced contingency with an unverified gate is a live bug, not a hedge. Also: SwiftUI `focusedSceneValue` closures must read state live at invocation — by-value captures of selection are stale by design, since republication lags.
